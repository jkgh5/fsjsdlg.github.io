03. Spark Clusters

Start a Spark Cluster and deploy a Spark application Heading

Run a Spark Cluster and Deploy Spark Applications

ND029 DSND C2 L1 A03 Spark Clusters V2

Spark Component Inventory

  • **Zookeeper: **connects to the Spark Master
  • spark-submit command: deploys the Spark Application
  • **Spark Master: **connects to the Spark workers.
  • **Spark Workers: **connect to the Spark Master on the Spark URI in a Standalone Cluster; in a Kubernetes configuration they are orchestrated by Kubernetes.

Starting the Spark Cluster

  1. Run the start-master.sh script
  2. Watch for the log file location
  3. Cat or tail the log file to find the Spark Master URI
  4. Run the start-slave.sh script passing the Spark Master URI parameter

Spark Startup Commands

$ /data/spark/sbin/start-master.sh
$ tail -f /opt/spark-2.3.4-bin-hadoop2.7/logs/spark--org.apache.spark.deploy.master.Master-1-719f72471d5b.out
$ /data/spark/sbin/start-slave.sh spark://719f72471d5b:7077

Multi-Language Support

  • You can write Spark Streaming applications using R, which is very popular for Data Science applications
  • You can also use Java, which is very familiar to a large portion of the Development Community
  • Spark also supports Scala, one of the foundational languages for Spark and Big Data
  • Lastly, Spark supports Python, one of the most popular programming languages

Data Streaming constructs

  • R uses DataFrames
  • Java and Scala use typed DataSets
  • Python uses DataFrames

Why use Python?

  • Python is not strictly typed
  • Popular for Data Science applications

What is a Data Pipeline?

**Data Pipeline: **a series of processing steps that augment or refine a raw data source in preparation for consumption by another system.

Structure of a Data Pipeline

  1. Every Spark Application starts by reading in a source
  2. A series of steps is executed that transforms the original flow of information
  3. The end result is a DataFrame that can be sinked externally with an outside system

The source is continually polled for updates to apply the transformations.* This continual polling is what sets Spark Streaming apart from non-streaming applications.*

Sample Spark Streaming Application

spark = SparkSession.builder.appName("balance-events").getOrCreate()

kafkaRawStreamingDF = spark                          \    
.readStream                                          \
.format("kafka")                                     \
.option("kafka.bootstrap.servers","localhost:9092)   \
.option("subscribe","balance-updates")               \
.option("startingOffsets","earliest")                \
.load()

kafkaStreamingDF = kafkaRawStreamingDF.selectExpr("cast(key as string) key,"cast(value as string) value")

kafkaStreamingDF.writeStream.outputMode("append").format("console").start().awaitTermination()

Explanation of Sample Spark Streaming Application

Note: the import of SparkSession is omitted for brevity

  • We start by instantiating a Spark session- this connects us to the Spark cluster
  • Next, we open a streaming source (Kafka in this case)
  • Third, we apply data transformations
  • Last we sink data to something external to Spark (the console is a good sink to test your data quality visually)

From Start to Deployment

To take advantage of the resources in the Spark cluster, we submit (or deploy) the Spark code:

  1. Start the Spark master
  2. Start the Spark workers
  3. Submit your application
$ /home/workspace/spark/sbin/start-master.sh
$ /home/workspace/spark/sbin/start-slave.sh spark://719f72471d5b:7077
$ /home/workspace/spark/bin/spark-submit /home/workspace/hellospark.py

If you ever need to stop the master, the script for that is:

$ /data/spark/sbin/stop-master.sh

If you ever need to stop the worker, the script for that is:

$ /data/spark/sbin/stop-slave.sh

Streaming Results

  • When your spark streaming application begins producing data, it will push the data to the sink you have created
  • By default, Spark executes in micro-batches
  • When sinking to the console you will see the batches
  • When sinking to anything else, you won't see the batches on the console
  • When sinking to the console, you should wait up to 2 minutes to see the results
  • When you see Batch 0 you have started getting results
